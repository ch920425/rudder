# In-app UI Lab

## Context

Rudder already has two internal UI reference surfaces:

- `/design-guide`, implemented by `ui/src/pages/DesignGuide.tsx`
- `/tests/ux/runs`, implemented by `ui/src/pages/RunTranscriptUxLab.tsx`

The current design guide is valuable, but it is a long single page with static
coverage badges rather than a navigable component lab. The transcript UX lab is
fixture-backed and useful, but it is separate from the broader component
inventory.

## Decision

Build on the existing in-app surfaces instead of introducing Storybook first.
The app already has the real theme, route handling, organization prefixing,
component CSS, and product-context constraints that component examples need.

Canonical route:

- `/ui-lab` becomes the main internal lab route.
- `/design-guide` stays compatible by rendering the same lab.
- `/tests/ux/runs` remains compatible by opening the transcript section.

## Component Inventory Model

The UI Lab inventory tracks renderable components and product patterns, not
every file under `ui/src/components`.

In scope:

- exported primitives from `ui/src/components/ui`
- reusable product components that can render with fixture data
- fixture-backed workflow surfaces such as run transcripts
- common product patterns from the current design guide

Out of scope for direct rendering:

- test files
- pure helpers, types, and constants
- context-bound containers that require live server data
- full page routes that are better verified through feature E2E flows

Coverage statuses:

- `covered`: rendered directly in the lab
- `fixture-backed`: rendered through stable fixture data
- `context-required`: listed, but needs live app context or API data
- `not-renderable`: helper, type, shell, or non-visual module
- `deprecated`: retained only for compatibility
- `missing-example`: in scope but still needs a lab example

## Contributor Workflow

When adding or materially changing a common UI primitive, reusable product
component, or product pattern:

1. Add or update a UI Lab example when the component can render with fixtures.
2. Add a coverage inventory row even when the component is context-bound.
3. Mark the row honestly instead of forcing a broken placeholder.
4. For visible workflow changes, keep E2E and browser verification as the source
   of truth; the lab is a review aid, not a replacement for workflow tests.

## Phase 1 Scope

- Add the `/ui-lab` shell with section navigation and coverage filtering.
- Preserve the current `DesignGuide` examples inside the lab.
- Add a common components gallery for high-use product components.
- Add a coverage registry with explicit status and gaps.
- Make the run transcript UX lab reachable as a lab module.
- Keep legacy routes compatible.

## Phase 1 Acceptance

- `/ui-lab` renders and is organization-route aware.
- `/design-guide` still reaches the same lab content.
- `/tests/ux/runs` opens the transcript lab section.
- Search/filter works against the coverage registry.
- Current design guide examples remain available.
- Common components are visible with fixture-backed examples.
- Browser verification covers desktop and narrow viewport.
- Typecheck and targeted tests pass.

## Non-goals

- Public documentation site changes.
- Storybook or new build tooling.
- Backend, schema, API, or organization data model changes.
- Exhaustively rendering every context-bound page in phase 1.
