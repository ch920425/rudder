# Agent Learning And Skill Improvement

## Summary

Rudder should let operators teach agents from real runs, and let agents periodically propose improvements from their own execution history. The product concept should be **Agent Learning**, not SOP generation.

The user's mental model is simple:

```text
feedback on real work -> agent learns -> learning becomes a skill update -> future runs load that skill
```

The runtime model stays equally simple:

```text
run evidence -> feedback batch -> learning draft -> skill update proposal -> approved skill revision -> agent skill binding -> next run
```

This proposal replaces the previous SOP framing. SOP should not be a user-facing object. At most, a "procedure" can be one section inside a generated skill. The durable object that changes agent behavior is an Agent Skill revision.

## Product Decision

Use **Agent Learning** as the top-level product language.

Do not introduce a separate SOP library, SOP generator, or SOP rollout surface. Those add a second governance object that users must understand before they can understand the thing that actually affects runtime behavior.

Users should see:

- `Teach this agent`
- `Leave feedback on this run`
- `Review learning`
- `Review skill update`
- `Enable for this agent`
- `Learning applied in this run`

Rudder should store and govern:

- feedback and evidence from runs
- learning drafts generated from that evidence
- structured skill update proposals
- immutable skill revisions
- agent-to-skill bindings and revision policies
- run-level loaded skill revision metadata

## Why This Matters

Rudder's north-star metric is the weekly count of real agent-work loops completed end-to-end. A loop is more valuable when the next loop gets better because the system retained what was learned.

Today, repeated agent failures are easy to inspect but hard to convert into runtime improvement. A user may notice:

- the agent started coding before checking out the issue
- the agent repeated the same context reads across runs
- the handoff did not explain owner, reason, or remaining work
- a run ended without a clear issue comment or done signal
- an agent consistently succeeds when it follows a repo-specific verification sequence

If that feedback stays in comments or chat, it does not reliably change future runs. If the user must directly edit prompts or raw skill files, the interaction is too low-level.

The missing product loop is:

```text
observe work -> teach from evidence -> approve a skill improvement -> measure future behavior
```

## Design Principles

1. Keep the user-facing model small.
   Users teach agents and review learning. They should not need to understand an intermediate SOP artifact.

2. Use skills as the execution substrate.
   Agent Skills are already the right runtime abstraction for reusable know-how, workflows, guardrails, examples, and evaluation checklists.

3. Make learning reviewable, not silent.
   Agents may propose improvements from past work, but they must not silently mutate production behavior.

4. Store structured learning, render readable Markdown.
   Human-readable Markdown is not enough for traceability, diffing, conflict detection, or evaluation. Each approved skill revision needs a structured spec with stable rule/check identifiers.

5. Keep every object organization-scoped.
   Feedback, evidence links, learning drafts, skill proposals, revisions, bindings, evaluation reports, and loaded-skill metadata all need explicit `org_id` scoping.

6. Make runtime behavior reproducible.
   Every run should be able to answer which skill revision was loaded, not just which skill key was enabled.

## Core Concepts

### Run Feedback

Run Feedback is the user's or system's judgment about a specific piece of execution. It can point to a transcript block, run event, tool call, log range, issue comment, work product, final output, approval event, or run summary.

Feedback can be:

- manually written by the operator
- system-derived from run status, missing closeout, eval failure, approval rejection, budget pause, or tool error
- proposed by an agent as a self-report, without being accepted as final truth

Feedback is evidence input. It does not directly change runtime behavior.

### Feedback Batch

A Feedback Batch is a durable set of feedback items submitted together for a target learning goal.

Examples:

- "Teach this engineer agent how to close out issue work."
- "Improve the code-review skill using these three review failures."
- "Capture the verification sequence that worked in this successful run."

The batch records submitter, target agent, target skill if known, source runs, source issues, feedback items, and intended scope.

### Learning Draft

A Learning Draft is Rudder's proposed interpretation of feedback. It says what the agent should learn, why, when it applies, and which evidence supports it.

A learning draft is not a standalone runtime object. It is a review surface that leads to a skill update proposal.

### Skill Update Proposal

A Skill Update Proposal is a reviewable patch against an Agent Skill.

It includes:

- target skill or proposed new skill
- base skill revision
- Markdown preview
- structured spec diff
- evidence links
- risk level
- expected impact
- eval and approval status
- rollback plan

### Skill Revision

A Skill Revision is an immutable approved version of a skill. It contains both rendered `SKILL.md` content and a structured spec.

The rendered Markdown is for agents and humans. The structured spec is for Rudder governance:

- stable rule ids
- applies-when predicates
- required and forbidden behaviors
- severity
- evidence links
- evaluation checks
- conflict metadata
- source proposal
- content hash

### Agent Skill Binding

An Agent Skill Binding says which agent loads which skill and which revision policy applies.

For the first version, support:

- one agent
- one skill
- `latest_approved` or `pinned_revision`

Role, org, experiment, or canary rollout can come later.

### Auto Learning Review

Auto Learning Review is a periodic reflection flow that reads recent runs, feedback, outcomes, and loaded skill revisions, then proposes skill updates.

It must create proposals, not silently apply them.

## User Journey A: Teach From A Run

1. The operator opens Run Detail.
2. The default view remains human-readable timeline first, with raw transcript and run events available through progressive disclosure.
3. The operator selects a message, tool card, event, output, issue comment, or run summary.
4. They click `Teach this agent`.
5. A feedback tray opens. Rudder automatically records durable evidence locators:
   - `run_id`
   - `agent_id`
   - `issue_id`
   - source kind
   - event id and sequence when available
   - log ref and byte range when available
   - transcript entry key when available
   - selected text snapshot
   - content hash
   - redaction version
6. The operator writes natural-language feedback:

   ```text
   Before changing code for an assigned issue, check out the issue and confirm ownership.
   Do not start modifying files without an in-progress issue assignment.
   ```

7. The operator can add multiple feedback items from the same run.
8. They submit the batch.
9. Rudder generates a learning draft and recommends a skill target.
10. The operator reviews the proposed skill update.
11. Approval creates an immutable skill revision and optionally enables it for the current agent.

## User Journey B: Review Auto Learning

1. An agent completes several runs over time.
2. Rudder periodically groups recent evidence:
   - repeated failure modes
   - repeated successful patterns
   - missing closeout signals
   - repeated unnecessary context reads
   - recurring approval or handoff gaps
   - skill usage that correlates with success or failure
3. Auto Learning Review creates a draft:

   ```text
   Engineer agent has 4 recent issue runs without a clear closeout comment.
   Proposed learning: strengthen the issue execution skill with explicit closeout rules.
   ```

4. The operator sees the review in an Agent Learning queue.
5. The operator can dismiss it, edit it, request more evidence, or create a skill update proposal.
6. No runtime behavior changes until approval.

## User Journey C: Review A Skill Update

The review page shows four sections.

### 1. Evidence

The operator sees source runs, issues, feedback items, selected snippets, system-derived signals, and confidence.

Each important proposed rule links back to original evidence.

### 2. Learning Summary

Rudder explains what changed in operational language:

```text
This update teaches the agent to always leave one of four closeout signals:
done, progress comment, blocker comment, or handoff comment.
```

### 3. Skill Diff

The operator sees a Markdown diff for `SKILL.md` and a structured diff for rules/checks.

Example structured rule:

```json
{
  "id": "issue-closeout-signal-required",
  "appliesWhen": {
    "runScene": "heartbeat",
    "issueBacked": true
  },
  "instruction": "Before ending an issue-backed run, leave a done, progress, blocker, or handoff signal.",
  "severity": "required",
  "evaluation": {
    "check": "issue_comment_or_status_change_exists",
    "window": "run"
  }
}
```

### 4. Enablement

The operator chooses:

- save revision but do not enable
- enable for this agent
- pin this agent to this revision

Future scopes are out of MVP:

- all agents with same role
- issue label or workflow type
- project
- organization-wide rollout
- experiment/canary rollout

## User Journey D: Observe Learning Applied

Run Detail should show a compact `Learning applied` area:

- loaded skills
- loaded skill revision ids or hashes
- whether the run matched any learning checks
- obvious deviations or missing signals

This should not require a full evaluator in the first version. The minimum useful display is:

```text
Loaded skills:
- issue-execution, rev 3, latest approved
- code-review, rev 1, pinned
```

Later versions can add adherence reports and suggested follow-up feedback.

## Agent Journey

### Worker Agent

The Worker Agent runs normal work. It receives enabled skill entries in runtime context and follows the loaded `SKILL.md`.

The Worker Agent does not need to understand feedback batches or learning drafts. It only needs clear skills at runtime.

### Learning Agent

The Learning Agent reads feedback batches, run evidence, and recent outcomes. It creates learning drafts and skill update proposals.

It must:

- cite source evidence
- propose narrow scopes when evidence is sparse
- detect conflicts with existing skill rules
- output both Markdown and structured spec changes
- send proposals to review instead of applying them

### Evaluator Agent

The Evaluator Agent checks whether a proposed update is coherent and whether future runs appear to follow it.

For MVP this can be minimal:

- validate structured spec shape
- verify evidence links resolve
- detect obvious conflicts
- record loaded skill revision per run

Later it can run richer adherence checks, replays, benchmarks, and canary comparisons.

### Manager Agent

For broader or risky changes, a Manager Agent can review the proposal before human approval.

This is optional in MVP. The important invariant is that agents can propose but cannot silently promote behavior-changing updates.

## End-To-End Flows

### Flow 1: Manual Feedback To Skill Revision

```text
Run Detail selection
-> feedback item
-> feedback batch
-> learning draft
-> skill update proposal
-> user review
-> approved skill revision
-> enable for selected agent
-> next run loads skill revision
-> run records loaded revision metadata
```

### Flow 2: Periodic Auto Learning Review

```text
recent runs
-> outcomes and feedback
-> pattern detection
-> learning review draft
-> skill update proposal
-> eval/approval gate
-> approved skill revision
-> optional enablement
```

### Flow 3: Skill Revision To Runtime

```text
organization skill
-> immutable skill revision
-> agent skill binding
-> runtime context resolves selected revision
-> adapter loads materialized skill files
-> heartbeat run stores loaded skill revision metadata
```

### Flow 4: Future Evaluation Loop

```text
run with learned skill loaded
-> basic checks or evaluator report
-> new feedback on deviations
-> new learning draft
-> skill revision patch
-> rollback or promote
```

## Information Architecture

### Run Detail

Add `Teach this agent` feedback actions to timeline blocks, tool cards, events, outputs, and run summaries.

Add a feedback tray for the current draft batch.

Add a compact `Learning applied` section showing loaded skill revisions.

### Agent Detail

Add a `Learning` section or tab that shows:

- recent feedback
- auto learning reviews
- open skill update proposals
- enabled skills and revision policies
- learning history

This can live near the existing Skills tab or become a subsection of it. The UX should reinforce that learning changes skills.

### Skill Detail

Show:

- current approved revision
- revision history
- evidence-backed changes
- agents using this skill
- direct edit policy

For generated/managed learning skills, direct edits should create a draft skill update proposal rather than mutating the approved revision in place.

### Agent Learning Queue

Optional after MVP. A board-level queue can aggregate:

- auto learning reviews
- pending skill update proposals
- failed eval gates
- stale or low-performing learned skills

## Skill Materialization

Approved learning becomes a skill revision.

Generated `SKILL.md` frontmatter should include:

```yaml
name: issue-execution
description: Follow the learned issue execution workflow for ownership, context, work chunks, closeout, blocking, and handoff.
rudder_managed: true
managed_by: agent_learning
skill_revision_id: <uuid>
skill_revision: 3
content_hash: <sha256>
source_proposal_id: <uuid>
```

Body sections should include:

- When to use this skill
- Learned workflow
- Required behavior
- Do not do
- Closeout rules
- Blocked and handoff rules
- Examples
- Evaluation checklist
- Evidence notes

The structured spec should be stored next to the revision, not inferred from Markdown after the fact.

## Direct Edit Policy

Existing organization skills may remain directly editable.

Managed learning skills need stronger rules:

1. Approved revisions are immutable.
2. Editing the current generated `SKILL.md` creates a draft skill update proposal.
3. Direct workspace edits should not silently alter the approved revision loaded by agents.
4. The Skill Detail page should show whether a skill is managed by Agent Learning.
5. If a generated skill is manually forked, Rudder should record that lineage and stop treating the fork as automatically governed by the old learning evidence.

This prevents direct edits from bypassing approval, evidence, and rollback semantics.

## MVP Scope

The first shippable version should prove the smallest useful loop.

MVP includes:

1. Single-run feedback tray with multiple feedback items.
2. Durable evidence locators for run events, log ranges, transcript entries, and selected text snapshots.
3. Feedback Batch submission.
4. Learning Draft generation from one batch.
5. Skill Update Proposal targeting one existing or new organization skill.
6. Structured spec with stable rule/check ids.
7. Markdown skill preview and diff.
8. User approval.
9. Immutable skill revision creation.
10. Manual enablement for one agent.
11. Runtime loading of the selected skill revision.
12. Run-level loaded skill revision metadata.
13. Basic `Learning applied` display on Run Detail.

MVP does not include:

- SOP library or SOP detail pages
- role-level or org-level rollout
- automatic production promotion
- complex A/B tests
- marketplace sharing
- cross-organization learning
- fully general evaluator agents
- user-custom generation prompts
- direct mutation of agent prompts outside skill revisions

The first version may include a manually triggered `Review recent runs` action for one agent. Scheduled periodic Auto Learning Review can land after the evidence and revision model is stable.

## Later Versions

### V1.1: Auto Learning Review

Add scheduled reviews for recent runs. The system detects repeated failures and repeated success patterns, then creates learning drafts.

### V1.2: Better Evaluation

Add structured adherence checks, benchmark links, and before/after comparisons for skill updates.

### V1.3: Scoped Rollout

Add role-level, issue-label, project, and organization rollout with approval gates.

### V1.4: Canary And Rollback

Allow a skill revision to run for a limited scope or fixed number of runs, compare against prior revision metrics, and recommend rollback if outcomes regress.

### V2: Shared Learning Library

Support reusable learning templates and curated skills that can be imported across organizations without exposing private run evidence.

## Success Metrics

Measure learning loops, not document production.

Primary metrics:

- weekly feedback-to-skill-revision loops completed
- auto learning reviews accepted by the operator
- skill update proposal approval rate
- time from feedback submission to approved skill revision
- number of runs with loaded skill revision metadata
- repeated failure rate after skill revision enablement
- closeout missing rate after issue-execution learning
- unnecessary context/tool call rate after learning
- rollback rate
- operator edits per generated learning draft

The strongest product signal is that real agent-work loops complete more reliably after feedback is converted into skills.

## Risks And Mitigations

### Risk: Skill Bloat

If every minor feedback item becomes a new skill, agents will load too much instruction and behavior will become noisy.

Mitigation: prefer updating existing skills, require scope, confidence, and evidence count, and show stale or low-use learning.

### Risk: Overfitting To A Single Run

One bad run can produce overly narrow or wrong guidance.

Mitigation: sparse evidence should produce narrow single-agent learning, not broad rollout. The UI should show evidence count and confidence.

### Risk: Black-Box Self-Modification

Users may lose trust if agents appear to rewrite themselves silently.

Mitigation: agents can propose learning, but approval creates the behavior-changing skill revision.

### Risk: Markdown-Only Governance

Markdown is readable but insufficient for diffing, conflict checks, and evaluation.

Mitigation: store structured spec alongside rendered skill Markdown.

### Risk: Direct Edits Bypass Review

If managed skills can be edited in place, approved evidence-backed behavior becomes unauditable.

Mitigation: generated learning skills are revisioned; edits create proposals or forks.

### Risk: Evidence Links Rot

Transcript UI may change, logs may be redacted, and event rendering may evolve.

Mitigation: store durable event ids, event sequence, log byte ranges, snapshots, hashes, and redaction version.

## Recommended Product Copy

Use learning language:

- `Teach this agent`
- `Leave feedback`
- `Create learning`
- `Review learning`
- `Proposed skill update`
- `Approve skill revision`
- `Enable for this agent`
- `Learning applied`
- `Review recent runs`

Avoid SOP language:

- `SOP Generator`
- `SOP Library`
- `SOP Rollout`
- `SOP Adherence`

The clean product promise is:

> Agents learn from feedback and past work. Rudder turns that learning into reviewable, versioned skills.

That keeps the user model simple while preserving the control-plane guarantees Rudder needs: evidence, approval, versioning, runtime reproducibility, and rollback.
